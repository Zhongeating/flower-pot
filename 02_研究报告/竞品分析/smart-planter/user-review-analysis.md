### 3.2 用户对「桌面有植物」的真实语言

**用户自己用的词**（按出现频率排序）：
- 「搭子」/「陪伴」/「陪着我」（25+次）
- 「可爱」/「萌」/「好看」（20+次）
- 「情绪」/「治愈」/「小快乐」（10+次）
- 「宠物」/「宝宝」/「小祖宗」/「盆友」/「精灵」（12+次）
- 「理直气壮」/「老板来了都说…」（3次）
- 「i人」/「社恐」（3次）

**词汇落在六维光谱的分布**：
| 维度 | 出现频次 | 占比 |
|------|:--:|:--:|
| 陪伴交互 | 25+ | ~40% |
| 感官精神提升（好看/治愈） | 20+ | ~30% |
| 休息的正当理由 | 5+ | ~8% |
| 仪式感 | 4+ | ~6% |
| 掌控/成就感 | 3+ | ~5% |
| 连接他人 | 7+ | ~11% |

> 「连接他人」的占比被送礼场景拉高，但送礼中的"连接"是送礼者和收礼人之间的连接，不是花盆本身的连接功能。

**画像汇总中未覆盖的新需求维度**：
- 🆕 **「不确定性消除」**（B-017）——不是懒，是不知道该不该做
- 🆕 **「数据→情绪翻译层」**（B-029）——显示数据没用，显示表情有用
- 🆕 **「甩锅机制」**（B-038）——花盆替用户承担失败责任
- 🆕 **「品类锚定：解压神器」**（B-048）——竞争对手不是花盆，是自动猫砂盆
- 🆕 **「桌面信息终端」**（B-019）——花盆替代时钟/温湿度计/to-do list
- 🆕 **「产品异化」**（B-043）——如果只关心表情不关心植物，产品就死了

### 6.5 对「植间」产品的关键启示

#### ✅ 方向确认——以下方向已被103条竞品评论强验证

1. **陪伴 > 功能**：用户买Ivy不是为了监测湿度，是为了「桌上有个人陪我」。我们的产品定义中「陪伴」必须是核心叙事，养护是手段不是目的。
2. **「养不死」是最强差异化**：Ivy被骂的核心原因是花盆在杀死植物（244lux、不透气、传感器失灵→错误浇水→烂根）。我们的浸盆式潮汐灌溉（注水→浸泡→抽回干湿循环）+补光灯+导风口直接回应所有致命缺陷。
3. **开放结构+补光灯=不需要窗台**：A-043「窗台没插座来回倒腾」的痛点在我们方案中不存在——工位插电即可。
4. **HA/MQTT生态联动是现成答案**：4条评论追着Ivy要米家/HA，我们已定的HA MQTT模式恰好是答案。
5. **i人/社恐是真实用户群**：从3个不同角度被用户自我认领。
6. **送礼场景是增长引擎**：20条送礼评论，含复购信号。

#### ⚠️ 需要警惕的风险

1. **「盆是宠物 vs 植物是主角」的张力尚未解决**：B-039「盆比植物更像宠物」和IDEA-013「植物是大白」指向不同方向。需要验证。
2. **B-031充电悖论必须避开**：如果我们的花盆也需要频繁充电，「植物能不能活全看我记不记得充电」会原封不动回给我们。
3. **B-043产品异化风险**：如果表情和植物状态脱钩，用户只关心表情不关心植物，「智能花盆」退化为「电子宠物」。
4. **不要叫「智能花盆」**：N2 12条（含 B-013/B-014/B-015/B-020 等 10 条负面重定义 + 4 条正面重定义）——「智能」标签正在引发系统性信任反噬。考虑品类命名为「桌面情绪终端」「有生命的桌面搭子」，品牌名「植间」已经走在正确方向上。
5. **定价如果超过¥300，准备好回答「这为什么不买Ivy」**：用户会用Ivy做价格锚点。

#### 🆕 可以融入产品设计的新洞察

1. **翻译层设计**（B-029）：屏幕不显示「湿度65%」，显示「😊我喝饱了」
2. **互动疲劳机制**（B-011）：摸太多→花盆显示「😑」，停一会→恢复「😊」
3. **随机惊喜机制**（A-044/B-033）：80%预期反馈+20%惊喜反馈
4. **日常仪式嵌入**（B-024）：上班逗花盆、浇水打卡、工作计划养成化
5. **人际关系连接**（A-025）：情侣共养、朋友间互相看到对方花盆状态
6. **「甩锅」叙事**（B-038）：产品话术——「养不好？那是上一代花盆的问题。」

### 6.6 用户评论的层级过滤：什么是真问题、什么是过度反应

> PM 核心能力：不是照单全收用户抱怨，而是区分哪些抱怨对应的是一旦不解决产品就死的"咽喉问题"，哪些是"改了更好不改也能活"的 NTH。

#### 🔑 过滤原则

一条用户评论是否构成**真问题**，用两个问题过滤：

1. **如果这个问题不解决，植物会死吗？** → 如果不会死，不紧迫。
2. **这个问题影响了用户的购买决策吗？** → 用户是在"不买"还是"买了但吐槽"？

#### 🚨 真问题：不解决产品不成立（植物会死 / 用户不会买）

| 问题 | 为什么是真问题 | 我们的回应 |
|------|------|------|
| **工位光照<300lux → 植物必死**（A-054） | 244lux 连苔藓都活不了——这不是体验问题，是物理定律。工位光照不足≠用户懒，是办公室建筑决定的 | ✅ 全光谱补光灯环 = 不需要自然光 |
| **封闭设计不透气→烂根**（A-054） | 植物根系需要呼吸。封闭花盆 + 闷湿 = 烂根，任何传感器都救不了 | ✅ 开放结构 + 主动导风口 |
| **传感器不准导致用户做了错误的浇水动作**（A-014） | 系统报渴但湿度100% → 用户继续浇水 → 淹死植物。传感器不准的后果不是"体验差"，是植物死亡 | ✅ 浸盆式潮汐灌溉采用定时干湿循环 + 湿度传感器辅助调度（非实时决策）。即使传感器偶有偏差，也不会出现"报口渴但实际很湿仍浇水"的矛盾——因为每次都是完整的"注水→浸泡→排干"循环，过量水分会被抽回 |
| **植物随机发+环境不匹配**（A-010） | 办公室暗→收到要晒太阳的植物→植物必死。用户没选错，是系统替他选错了 | ⚠️ 待定：我们是否绑定特定植物？如果绑定，必须确保植物匹配目标环境 |
| **信任崩溃（"骗人""伪智能"）**（B-013/B-015） | 如果用户认为"智能花盆"是一个虚假品类，就不会进入购买决策 | ✅ 品牌名「植间」绕过"智能"标签 |

#### 🟡 NTH（Nice to Have）：改了更好，不改植物也不会死

| 问题 | 为什么不是咽喉问题 | 判断 |
|------|------|------|
| **一天一充/必须插电**（A-003/A-019/A-031/B-031） | 用户吐槽"费电""续航差"，但**花盆插着电 ≠ 植物会死**。Ivy 的植物本身好养（喜阴耐湿），断电后只是表情没了，植物仍然活着。这属于"体验退化"而非"功能失效" | 🟡 我们已定**插电使用**（补光灯 + 蠕动泵需要电力），不需要为了"不插电"牺牲其他。但要确保：断电后植物不会立刻死 → 浸盆式潮汐灌溉的水箱可缓冲数日 |
| **表情花样太少**（A-002） | 用户玩腻了，但玩腻了 ≠ 产品死了。花盆的第一任务是保活植物，第二任务才是取悦用户 | 🟡 如果有 OTA 更新表情包的机制就是加分，但不是 MVP 的咽喉 |
| **不能接米家/HA**（B-001/B-004/B-006/A-027） | 用户想要，但没有米家 ≠ 植物会死 ≠ 产品不能卖 | 🟡 我们已定 HA MQTT 模式作为 L3 用户的功能，跑通了就是差异化；没做也不致命 |
| **关灯不熄屏**（A-008） | 刺眼，但用户可以把花盆转过去。不影响植物生存 | 🟡 光敏电阻 + 定时熄屏是后期 OTA 可加的 |
| **只能连2.4G WiFi**（A-004） | 办公室 5G 网络用户需要开热点——烦，但不致命 | 🟡 ESP32-S3 就是 2.4G，这是行业现状而非独有缺陷 |
| **价格太贵**（A-001/B-012/B-016） | 所有产品都有人嫌贵。关键不是价格，是**价格和用户感知价值是否匹配** | 🟡 等支付意愿访谈后再判断 |
| **APP 毫无必要**（A-009） | 一个人说"拎起盆就知道缺水"——这是真的，但也是一类用户的观点。另一类用户（A-022）说"APP数据挺准的，对小白友好" | 🟡 取决于目标用户。对 L1（全托型）APP 确实不重要；对 L3（深度控制型）APP 是核心 |

#### ⚠️ 看起来严重但实际不需要过度反应的问题

| 问题 | 为什么看起来严重 | 实际不需要过度反应的原因 |
|------|------|------|
| **B-031「植物能不能活全看记不记得充电」** | 黑色幽默，极具传播力 | 这句话的杀伤力在于传播，不是事实——Ivy 断电后植物仍然活着（自吸水），只是没了表情。**如果我们的产品也用同样逻辑，我们需要在传播层面回应这句话，但不需要在工程层面为"不插电"额外付出代价** |
| **B-043「关注的不是植物死活是表情」** | 听起来像产品异化 | 这是用户自嘲，不是预警。用户不会真的只关心表情——植物死了 = 花盆失去了存在意义。只要表情和植物状态有因果关联（健康→开心表情，缺水→难过表情），就不会脱钩 |
| **A-017「又被骗了买个垃圾」** | 非常情绪化的差评 | 这句话反映的是传感器失灵 + 客服敷衍的综合愤怒，不是对单个功能的否定。解决传感器可靠性后，这句话不会再出现 |
